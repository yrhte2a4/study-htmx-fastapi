# コミットメッセージルール

## 概要
Gitコミットメッセージの記述ルールを定義します。

## ルール

### 言語
- すべてのコミットメッセージは**日本語**で記述する

### フォーマット
コミットメッセージは以下の構造に従う：

```
<タイプ>: <簡潔な説明>

<詳細な説明（オプション）>

<フッター（オプション）>
```

### タイプ
以下のタイプを使用する：

- `feat`: 新機能の追加
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響しない変更（空白、フォーマット、セミコロンなど）
- `refactor`: バグ修正や機能追加を伴わないコード変更
- `perf`: パフォーマンス改善
- `test`: テストの追加や修正
- `chore`: ビルドプロセスやツールの変更

### 例

```
feat: ユーザー認証機能を追加

ログイン、ログアウト、セッション管理機能を実装。
JWTトークンを使用した認証方式を採用。

関連: #123
```

```
fix: タスク削除時のバグを修正

タスクを削除した際にローカルストレージが
正しく更新されない問題を解決。
```

```
docs: READMEにインストール手順を追加
```

### ベストプラクティス
- 簡潔な説明は50文字以内を目安にする
- 詳細な説明は必要に応じて追加する
- 何を変更したかではなく、なぜ変更したかを説明する
- 命令形ではなく、完了形で記述する

## コミット手順

### 推奨手順
コミットを実行する際は、以下の手順に従う：

1. **変更状況の確認**
   ```bash
   git status --porcelain
   ```

2. **変更内容の詳細確認**
   ```bash
   git --no-pager diff --cached
   ```
   - `--no-pager`オプションを使用してページャーを無効化
   - `--cached`オプションでステージされた変更のみを表示

3. **コミット実行**
   ```bash
   git commit -m "タイプ: 簡潔な説明

   詳細な説明（必要に応じて）

   関連情報（必要に応じて）"
   ```

### 注意事項
- `git status`や`git diff`がページャー（less）を使用する場合があるため、`--no-pager`オプションを活用する
- ステージされた変更を確認する場合は`--cached`オプションを使用する
- コミット前に必ず変更内容を確認し、意図した変更のみがコミットされることを確認する

## コミット実行制約

### 自動コミット禁止
- ユーザーから明示的なコミット指示がない限り、自動的にコミットを実行してはならない

## 適用範囲
このルールは常に適用されます（inclusion: always）。
